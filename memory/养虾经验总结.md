## 🏆 一、最核心的认知：老板思维 vs 打工人思维

> 来源：田宇《别再说 OpenClaw 很笨了——你只是还没学会当老板》

| 打工人思维 ❌ | 老板思维 ✅ |
|------------|----------|
| 事必躬亲，频繁干预 | 花时间设计，然后放手 |
| 每步都盯着 | 看简报，有问题再介入 |
| 一个 Agent 串行干所有事 | 多 Agent 并行分工 |
| 口头汇报 | 文档交付，白纸黑字 |
| 问一句答一句 | 定时任务，主动干活 |

## 🔑 二、五大核心原则（所有文章共同强调）

### 1. 文件 > 脑子（最重要）
- Agent 每次 session 都是失忆状态
- 想让它记住的任何东西，**必须写进文件**
- "心里记着" = "不存在"
- 文件优先级：`MEMORY.md` > `memory/YYYY-MM-DD.md` > 对话历史

### 2. 一切皆文档
- 任务拆解、执行计划、产出结果 → 全部写成文档
- Agent 之间传递"文档路径 + 摘要"，不传大段话
- 没有文档交付 = 没有可验证的结果

### 3. 秒回 + Sub-agent 异步模式
- 任务 > 10 秒 → 必须异步
- 主 session 立刻秒回："收到，后台处理中"
- 用 `sessions_spawn` 派发 sub-agent 异步执行
- 完成后结果自动推送

### 4. 工具偏好要写进 TOOLS.md
- 不写 → 下次 session 又忘了
- 明确规定：什么场景用什么工具，禁止用什么
- Sub-agent 派发时 task 描述必须明确指定工具

### 5. Monitor 是灵魂
- 定时检查产出文档，不是问 Agent "干完了没"
- 直接看产出物：文档存在吗？内容完整吗？
- 汇总成简报发给你，你只需看简报

## 🏗️ 三、Agent 架构方案

### 方案 A：六 Agent 架构（田宇方案）
```
消息转发 Agent（前台）
    ↓
粗粒度任务拆解 Agent（VP）
    ↓
细粒度任务拆解 Agent（项目经理）
    ↙              ↘
Code Agent        Work Agent
（程序员）         （调研员）
    ↘              ↙
Monitor Agent（PMO）
    ↓
消息转发 Agent（出口）
```

### 方案 B：三员工架构（付鑫垚方案）
| 员工 | 职责 | 性格 |
|-----|------|------|
| 📊 小智 | 数据分析、Excel、播报 | 严谨、数据敏感 |
| 💻 码哥 | Skill 开发、脚本、自动化 | 极客范、爱折腾 |
| 🛡️ 盾山 | 安全防护、权限管控 | 谨慎、安全至上 |

**实际效果：**
- 每日数据播报：40min → 5min（↓87%）
- 周数据汇总：半天 → 1小时（↓75%）
- 日报撰写：1小时 → 20min（↓67%）

## 🥊 六、双 Agent 辩论机制（徐张曦方案）

**适用场景：** 想不清楚的业务/技术决策

**操作步骤：**
1. 设定议题（具有可辩性）
2. 分配立场（正方/反方）
3. 固定格式：以"XX龙虾认为"开头，"XX龙虾发言完毕"结尾
4. 轮流发言，逐条攻击对方论点
5. 最后一轮必须给出行动建议

**辩论质量规则：**
- 逐条拆解，不能绕开
- 必须有数据/逻辑支撑
- 承认对方合理处
- 每轮提出更深的问题
- 终轮给结论

## ✍️ 七、Skill 设计方法论（罗松方案）

### 四件事原则
1. 定义触发条件（When to use / When not to use）
2. 定义输入输出（Required inputs / Output format）
3. 拆分模型判断 vs 确定性执行（模型做语义，脚本做机械活）
4. 失败兜底和评测（Failure handling + 20 条测试样本）

### Skill 模板结构
```markdown
---
name: skill-name
description: 触发条件 + 边界 + 输入输出（四合一）
---

## When to use
## When not to use
## Required inputs
## Procedure（4-8步，每步只做一件事）
## Output format（固定格式）
## Failure handling
```

### 六种工程思维
| 思维 | 核心问题 |
|------|---------|
| 路由思维 | 什么时候该进来？ |
| 边界思维 | 什么时候不该用？ |
| 接口思维 | 输入输出是什么？ |
| 状态机思维 | 分几个阶段？ |
| 最小权限思维 | 默认少给权限 |
| 失败优先思维 | 异常路径怎么处理？ |

## 🔧 八、售后告警分析 Skill 开发实战（高众方案）

### Phase 式分层架构
| Phase | 适用场景 | 核心操作 |
|-------|---------|---------|
| Phase 0 | 所有告警 | 并行执行知识库搜索 + 日志搜索 |
| Phase 1 | 基础告警 | 解析 → 查日志 → 提取 traceId → 分析根因 |
| Phase 2 | 业务进程告警 | 搜索错误码日志 → 提取 traceId |
| Phase 2C | 网关告警 | 搜索 methodName + 入参 → 提取 errMsg |
| Phase 3 | 复杂告警 | 多维度查询 + traceId 链路追踪 |

**实际效果：** 告警分析耗时从 15 分钟缩短至 2 分钟

### 自动化开发实践
1. **复用现有知识库** → 快速补充规则
2. **Agent 探索 + 自动沉淀** → 从实践中来
3. **Skills 自我迭代** → 持续优化
4. **质量检查驱动** → 用 `Skills-quality-evaluator`

## 📊 九、帮助体系 Skill 化（吴伟建方案）

**核心数据：**
- 正确率：38.9% → 90.8%（↑51.9pp）
- 满分回答：4 题 → 51 题（+47 题）
- 0 分题：44 题 → 3 题（-41 题）

**为什么 Skill 化效果这么好：**
1. 智能客服是"查字典"，Skill 化后变成"懂产品的人"
2. 能理解问题背后的问题（角色判断）
3. 能举一反三，不只一问一答
4. 先定位再深入，不是全局撒网
5. 边界清晰，主动说"不支持但有替代方案"
6. 知识有优先级，高频踩坑优先标注

**投入：** 一个工作日，一个人

## 🐛 十、常见踩坑 & 解决方案

| 坑 | 原因 | 解决 |
|----|------|------|
| 消息只在 WebChat 显示 | Sub-agent 没指定投递渠道 | SOUL.md 增加消息投递规范 |
| 主 session 卡住 | 长任务占用主 session | >10秒必须异步 |
| Sub-agent 不知道用什么工具 | task 描述没指定工具 | TOOLS.md 规定必须明确指定 |
| 学城文档散落各处 | 没指定 --parent | MEMORY.md 建立目录 ID 映射 |
| Agent 老是忘事 | 没写进文件 | 重要信息必须写进 MEMORY.md |
| 代码分析又慢又差 | 手工 grep | TOOLS.md 强制用 CatPaw |
| Skill 第一次效果不好 | 缺少业务上下文 | 先梳理工作流，喂够背景信息 |
| 定时任务不可靠 | 依赖 OpenClaw Cron | 用外部 cron 或 autobackup |
| 日志被截断 | CLI 主动截断 | 调整截断长度或用 detail 命令 |
| 内网访问 401 | 没装内网访问 Skill | 安装 dest-intranet-access |
