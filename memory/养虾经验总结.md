## 🏆 最核心的认知：老板思维 vs 打工人思维

> 来源：田宇《别再说 OpenClaw 很笨了——你只是还没学会当老板》

**用得最差的人，往往是最勤快的人。**

| 打工人思维 ❌ | 老板思维 ✅ |
|------------|----------|
| 事必躬亲，频繁干预 | 花时间设计，然后放手 |
| 每步都盯着 | 看简报，有问题再介入 |
| 一个 Agent 串行干所有事 | 多 Agent 并行分工 |
| 口头汇报 | 文档交付，白纸黑字 |
| 问一句答一句 | 定时任务，主动干活 |

## 🔑 五大核心原则（所有文章共同强调）

### 1. 文件 > 脑子（最重要）
- Agent 每次 session 都是失忆状态
- 想让它记住的任何东西，**必须写进文件**
- "心里记着" = "不存在"
- 文件优先级：`MEMORY.md` > `memory/YYYY-MM-DD.md` > 对话历史

### 2. 一切皆文档
- 任务拆解、执行计划、产出结果 → 全部写成文档
- Agent 之间传递"文档路径 + 摘要"，不传大段话
- 没有文档交付 = 没有可验证的结果

### 3. 秒回 + Sub-agent 异步模式
- 任务 > 10 秒 → 必须异步
- 主 session 立刻秒回："收到，后台处理中"
- 用 `sessions_spawn` 派发 sub-agent 异步执行
- 完成后结果自动推送

### 4. 工具偏好要写进 TOOLS.md
- 不写 → 下次 session 又忘了
- 明确规定：什么场景用什么工具，禁止用什么
- Sub-agent 派发时 task 描述必须明确指定工具

### 5. Monitor 是灵魂
- 定时检查产出文档，不是问 Agent "干完了没"
- 直接看产出物：文档存在吗？内容完整吗？
- 汇总成简报发给你，你只需看简报

## 🎯 多 Agent 协作配置（赵梦欢方案）

### 快速招人
```bash
openclaw agents add coder
openclaw agents add researcher
openclaw agents list --bindings
```

### 分配通道
```json
{
  "bindings": [
    { "agentId": "main", "match": { "channel": "daxiang" } },
    { "agentId": "researcher", "match": { "channel": "feishu" } }
  ]
}
```

### 开通内线通信
```json
{
  "tools": {
    "agentToAgent": { "enabled": true, "allow": ["main", "coder", "researcher"] },
    "sessions": { "visibility": "all" }
  }
}
```

### 共享白板
```bash
mkdir -p ~/.openclaw/shared/notes
```
```json
{
  "agents": {
    "defaults": {
      "memorySearch": { "extraPaths": ["~/.openclaw/shared"] }
    }
  }
}
```

## ✍️ Skill 设计方法论（罗松方案）

### 四件事原则
1. 定义触发条件（When to use / When not to use）
2. 定义输入输出（Required inputs / Output format）
3. 拆分模型判断 vs 确定性执行（模型做语义，脚本做机械活）
4. 失败兜底和评测（Failure handling + 20 条测试样本）

### Skill 模板结构
```markdown
---
name: skill-name
description: 触发条件 + 边界 + 输入输出（四合一）
---

## When to use
## When not to use
## Required inputs
## Procedure（4-8步，每步只做一件事）
## Output format（固定格式）
## Failure handling
```

### 六种工程思维
| 思维 | 核心问题 |
|------|---------|
| 路由思维 | 什么时候该进来？ |
| 边界思维 | 什么时候不该用？ |
| 接口思维 | 输入输出是什么？ |
| 状态机思维 | 分几个阶段？ |
| 最小权限思维 | 默认少给权限 |
| 失败优先思维 | 异常路径怎么处理？ |

## 🥊 双 Agent 辩论机制（徐张曦方案）

**适用场景：** 想不清楚的业务/技术决策

**操作步骤：**
1. 设定议题（具有可辩性）
2. 分配立场（正方/反方）
3. 固定格式：以"XX龙虾认为"开头，"XX龙虾发言完毕"结尾
4. 轮流发言，逐条攻击对方论点
5. 最后一轮必须给出行动建议

**辩论质量规则：**
- 逐条拆解，不能绕开
- 必须有数据/逻辑支撑
- 承认对方合理处
- 每轮提出更深的问题
- 终轮给结论

## 🐛 常见踩坑 & 解决方案

| 坑 | 原因 | 解决 |
|----|------|------|
| 消息只在 WebChat 显示，大象收不到 | Sub-agent 没指定投递渠道 | SOUL.md 增加消息投递规范，指定 `channel: "daxiang"` |
| 主 session 卡在长任务上 | 长任务占用主 session | SOUL.md 增加"秒回+Sub-agent"强制规范，>10秒必须异步 |
| Sub-agent 不知道用什么工具 | task 描述没指定工具 | TOOLS.md 规定 sub-agent 派发时必须明确指定 CLI |
| 学城文档散落各处 | 没指定 --parent 参数 | MEMORY.md 建立目录 ID 映射，创建时必须指定父文档 |
| Agent 老是忘事 | 没写进文件 | 重要信息必须写进 MEMORY.md 或 TOOLS.md |
| 代码分析又慢又差 | 手工 grep 逐个读文件 | TOOLS.md 强制规定代码任务走 CatPaw Claude Code |
| Skill 第一次效果不好 | 缺少业务上下文 | 先梳理自己的工作流，喂给 Skill 足够的背景信息 |
| 定时任务不可靠 | 依赖 OpenClaw Cron | 用外部 cron 或 openclaw-autobackup 替代 |

## 💡 ikun 可以立刻用的场景（服务端 RD 视角）

### 🔥 高价值场景

1. **PR Review 自动化**
   - 用 `sessions_spawn` 派 sub-agent 异步 review
   - 结合代码分析工具做深度 review
   - 产出结构化评审结论（P0~P3）

2. **线上告警分析**
   - `nocode-raptor-common-analyzer` skill（需内网）
   - 自动分析 Raptor 告警、logview 堆栈、trace 链路

3. **双 Agent 技术方案辩论**
   - 遇到架构选型、方案 PK 时
   - 让两个 Agent 从正反方辩论，逼近最优解

4. **定时日报/周报**
   - 每天 20:00 自动汇总工作内容
   - 用 cron 定时触发，不需要手动操作

5. **售后工单分析**（你的主业）
   - 批量分析 TT 工单，找规律
   - 自动生成售后问题分类报告

### 📋 中价值场景

6. **学城文档自动归档**
   - 建立目录结构，所有产出自动归档
   - 不再找不到文档

7. **会议室自动抢订**
   - 配置 cron 定时抢订
   - 指定楼层、时间、人数

8. **代码仓库分析**
   - 让 Agent 分析售后相关代码链路
   - 快速定位问题代码
