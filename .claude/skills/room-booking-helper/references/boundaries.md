# Quick Book vs Agent 流程分界线

本文档明确说明两种预订方式的适用场景和限制。

## Quick Book 脚本（快速方案）

### ✅ 适用场景

使用 Quick Book 当且仅当：

1. **所有参数都已明确指定** - 无需交互式确认
   - 城市、建筑、日期、时间、人数都已确定
   - 建筑名称能通过关键词唯一匹配
   
2. **单个建筑预订** - 一次预订一个会议室
   - 不需要多建筑查询或批量预订
   
3. **标准时间范围** - 符合以下约束
   - 单次预订时长 5 分钟 ~ 240 分钟（4 小时）
   - 预订日期在 8 天内（普通会议室）或 30 天内（培训会议室）

4. **无需高级筛选** - 只基于人数筛选
   - 不关心是否有投影仪、Zoom 等设备
   - 不关心是否靠窗户

### ❌ 不适用场景

以下任一情况都应该使用完整 Agent 流程：

1. **模糊建筑查询**
   - 用户说 "上海那个大楼" 或 "北京靠近地铁的办公室"
   - 建筑名称需要列表选择而非关键词匹配
   - **原因**：Quick Book 匹配失败或选错建筑
   - **解决方案**：Agent 交互式选择

2. **多建筑查询**
   - 用户说 "在上海各个建筑各给我预订一个"
   - 需要循环调用或批处理
   - **原因**：Quick Book 只处理单个建筑
   - **解决方案**：Agent 循环调用或新脚本 `batch_book.py`

3. **复杂时间需求**
   - 用户说 "周一到周五每天下午3点"（周期性预订）
   - 用户说 "从今天下午5点到明天上午10点"（跨天预订）
   - **原因**：Quick Book 只支持单日期 + 单时间段
   - **解决方案**：Agent 处理逻辑或 `recurring_book.py`

4. **带条件筛选的预订**
   - 用户说 "要有投影仪，靠窗，能容纳20人"
   - 需要按设备、窗户等条件筛选
   - **原因**：Quick Book 的 `cap_range()` 只按人数筛容量，不支持设备筛选
   - **解决方案**：修改 Quick Book 或使用 Agent

5. **备选方案查询**
   - 用户说 "明天3点都满了，给我列出4点的所有空闲房间"
   - 预订失败后需要看所有备选列表而非自动创建监测
   - **原因**：Quick Book 失败后只触发监测，不返回备选列表
   - **解决方案**：Agent 进行多时间段查询和推荐

6. **预订记录管理**
   - 用户说 "帮我取消周一下午3点的会议室预订"
   - 用户说 "修改上周四的预订时间"
   - **原因**：Quick Book 完全不支持取消/修改操作
   - **解决方案**：需要新增 `cancel_book.py` 或 `modify_book.py`

7. **权限和配额问题**
   - 同一个人同一间会议室每天最多预订3次
   - 用户试图超限预订
   - **原因**：Quick Book 预订失败但无提前提示
   - **解决方案**：Agent 先检查配额再提示用户

8. **跨天或长时间预订**
   - 用户说 "预订2小时，从今天下午5点到明天上午10点"
   - **原因**：Quick Book 检查到跨天直接拒绝
   - **解决方案**：API 支持但 Quick Book 不支持

9. **API 分页数据**
   - 某些接口在数据量很大时分页返回
   - Quick Book 可能只处理第一页数据
   - **原因**：缺少分页逻辑
   - **解决方案**：添加分页处理

10. **缓存失败自动重试**
    - 缓存 7 天后过期，用户在第 7 天用旧数据预订失败
    - **原因**：Quick Book 没有自动清除缓存重试的逻辑
    - **解决方案**：添加 "缓存命中失败后自动清除重试" 的逻辑

## 三种脚本架构

### 1️⃣ quick_query.py（仅查询）

**用途**: 列出所有空闲会议室，不自动预订

**参数**:
```bash
python3 quick_query.py \
  --city 上海 --building D2 \
  --date 2026-03-10 --start 12:00 --end 13:00 \
  [--capacity 5] [--floors 2层 3层] [--window EXIST] [--equips Zoom]
```

**返回值**:
```json
[
  {
    "index": 1,
    "id": 11486,
    "name": "富春厅",
    "building": "上海/互联D2",
    "floor": "2层",
    "capacity": 20,
    "window": "有窗",
    "equipment": ["Zoom", "投影仪"],
    "roomMap": "https://..."
  },
  ...
]
```

**适用场景**:
- ✅ 用户只是想查看有哪些空闲会议室
- ✅ 用户需要看多个备选方案后再选择
- ✅ Agent 进行信息查询而非直接操作
- ✅ 需要返回完整列表供后续处理

### 2️⃣ quick_book.py（快速预订）

**用途**: 参数完整就直接预订，无需确认

**参数**:
```bash
python3 quick_book.py \
  --city 上海 --building D2 \
  --date 2026-03-10 --start 12:00 --end 13:00 \
  --capacity 5 [--mis wangjun137]
```

**返回值**:
```
✅ 预订成功！会议室: 富春厅 (5-6人)
预订时间: 2026-03-10 12:00-13:00
```

**适用场景**:
- ✅ 所有参数都已明确指定
- ✅ 用户明确说"帮我订这个"
- ✅ 建筑名称能唯一匹配
- ✅ 无需交互确认

**何时不用**:
- ❌ 用户只是查询，未明确要求预订
- ❌ 建筑名称模糊需要列表选择
- ❌ 用户想看备选方案

### 3️⃣ 完整 Agent 流程（交互式）

**用途**: 支持自然语言对话，处理复杂需求

**适用场景**:
- ✅ 用户表述模糊，需要多轮确认
- ✅ 需要建筑列表选择
- ✅ 需要处理备选方案
- ✅ 需要取消/修改预订
- ✅ 周期性预订等复杂需求

**优势**:
- **交互式对话** - 支持自然语言理解和多轮确认
- **高级筛选** - 支持设备、窗户、容量等多条件组合
- **智能推荐** - 预订失败时自动推荐备选方案
- **完整管理** - 支持查看、取消、修改预订
- **灵活处理** - 处理各种边界情况和异常

## 📋 使用决策树

```
用户提出需求
    ↓
是否明确表示"查询"或"看看有哪些"？
    ├─ 是 → 使用 quick_query ✅（列出所有会议室）
    │       ↓
    │   用户看到列表后再决定是否预订
    │
    └─ 否 → 是否明确说"帮我订"或"预订这个"？
            ├─ 否 → 不确定意图，先用 quick_query 查询
            │
            └─ 是 → 所有参数都已明确？
                    ├─ 否 → 使用 Agent（需要交互式收集信息）
                    │
                    └─ 是 → 建筑名称能唯一匹配？
                            ├─ 否 → 使用 Agent（需要列表选择）
                            │
                            └─ 是 → 有高级筛选需求（设备等）？
                                    ├─ 是 → 使用 Agent
                                    │
                                    └─ 否 → 单个建筑且单次预订？
                                            ├─ 否 → 使用 Agent（多建筑等）
                                            │
                                            └─ 是 → 使用 quick_book ✅（直接预订）
```

## 参数对比表

| 功能 | Quick Book | Agent | 备注 |
|------|-----------|-------|------|
| 参数输入 | 命令行必填 | 交互式可选 | Agent 支持默认值和推断 |
| 建筑选择 | 关键词模糊匹配 | 列表交互选择 | 模糊时 Agent 更安全 |
| 时间范围 | 单日期 + 单时段 | 支持周期、跨天等 | - |
| 容量筛选 | ✅ 按人数 | ✅ 按人数 | - |
| 设备筛选 | ❌ | ✅ 投影、Zoom 等 | Quick Book 计划支持 |
| 窗户筛选 | ✅ | ✅ | - |
| 多建筑 | ❌ 单个 | ✅ 多个 | - |
| 查看预订 | ❌ | ✅ | - |
| 取消预订 | ❌ | ✅ | - |
| 修改预订 | ❌ | ✅ | - |
| 备选方案 | 自动创建监测 | 列出所有候选 | Agent 体验更好 |
| 配额检查 | ❌ 事后报错 | ✅ 事前提示 | - |

## 推荐流程

### 场景1：简单查询需求
```
用户："帮我看一下上海D2明天12-13点有没有空会议室"
      ↓
     quick_query（仅查询，不预订）
      ↓
   输出: 所有空闲会议室列表
      ↓
用户看到列表后："好的就订第一个吧"
      ↓
     quick_book --room-id xxx（指定会议室直接预订）
```

### 场景2：明确预订需求
```
用户："帮我订一个上海D2明天12-13点的5人会议室"
      ↓
     quick_book（所有参数明确，建筑能唯一匹配）
      ↓
   输出: 预订成功
```

### 场景3：需要交互的需求
```
用户："我明天下午需要一个会议室，在北京，有Zoom就最好了"
      ↓
     Agent（时间模糊，需要多轮确认）
      ↓
    "下午具体几点？15-16点可以吗？"
      ↓
用户："可以的"
      ↓
     quick_query（查询后显示列表）
      ↓
用户："就这个吧"
      ↓
     quick_book（最终预订）
```

### 场景4：复杂需求
```
用户："帮我查一下本周每天下午3点，北京和上海各建筑有没有空会议室"
      ↓
     Agent（周期性 + 多城市）
      ↓
   循环调用 quick_query 多次汇总结果
```

### 场景5：编辑操作
```
用户："取消周一下午3点的预订，改到周二"
      ↓
     Agent（需要取消/修改能力）
      ↓
   调用 cancel_book.py / modify_book.py（待实现）
```
