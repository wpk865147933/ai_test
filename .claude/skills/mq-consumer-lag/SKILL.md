# MQ 消费组积压排查 Skill

## When to use
- 用户提到"消费组积压"
- 用户说"MQ 消费慢了"
- 用户要求排查"消息堆积"
- 用户提供消费组名称

## When not to use
- 用户只是要看消费组列表
- 用户要创建/删除消费组

## Required inputs
- 消费组名称或 AppKey
- （可选）Topic 名称
- （可选）预期积压阈值

## Procedure

### Phase 1: 确认积压情况
通过 Mafka/Swan 等工具查询：
```bash
# 查看消费组状态（需要相应的 CLI 工具）
# 或通过监控平台 API 获取

# 关键指标：
# - Lag（积压量）
# - Consume Rate（消费速率）
# - Produce Rate（生产速率）
# - Consumer Instance Count（消费者实例数）
```

### Phase 2: 定位根因

#### 情况 1: 生产过快
- 生产速率 > 消费速率
- 检查上游是否有异常流量
- 检查是否有批量推送任务

#### 情况 2: 消费过慢
- 检查消费者日志，是否有大量 ERROR
- 检查下游依赖（DB、RPC）是否响应慢
- 检查是否有消息处理超时
- 检查是否有死信消息卡住

#### 情况 3: 消费者异常
- 检查消费者实例数是否正常
- 检查是否有消费者重启/宕机
- 检查是否有消费者卡在某条消息

### Phase 3: 深度分析
对于复杂的积压问题：

1. **采样分析消息**：
   - 抽取积压的消息样本
   - 分析消息类型分布
   - 识别是否有特定类型的消息卡住

2. **链路追踪**：
   - 获取慢消息的 traceId
   - 通过 Cat/Raptor 追踪处理链路
   - 找到耗时最长的环节

3. **日志分析**：
   - 搜索消费失败的日志
   - 统计失败原因分布
   - 找到高频失败场景

### Phase 4: 输出排查报告
```markdown
## MQ 消费组积压排查报告

**消费组**: xxx
**Topic**: xxx
**当前积压**: xx 万条
**生产速率**: xx TPS
**消费速率**: xx TPS

### 📊 积压趋势
| 时间 | 积压量 | 消费速率 |
|------|--------|---------|
| xx:xx | xx | xx |
| xx:xx | xx | xx |

### 🔍 根因分析
**积压原因**: [生产过快 / 消费过慢 / 消费者异常]

**详细分析**:
1. xxx
2. xxx

**关键证据**:
- 日志片段：xxx
- 链路追踪：xxx
- 消息样本：xxx

### 💡 解决方案

#### 短期处理
1. [ ] 扩容消费者实例
2. [ ] 跳过/重试失败消息
3. [ ] 临时限流上游生产

#### 长期优化
1. [ ] 优化消费逻辑（xxx）
2. [ ] 增加异步处理能力
3. [ ] 设置合理的告警阈值

### 📈 监控建议
- 告警阈值：积压量 > xx 万条
- 监控指标：xxx
- 检查频率：xxx
```

## Output format
- Markdown 格式的排查报告
- 包含根因分析、解决方案、监控建议
- 关键信息用高亮标记

## Failure handling
1. 如果无法获取消费组信息 → 检查工具权限或登录状态
2. 如果积压原因不明确 → 建议查看日志和链路追踪
3. 如果是下游系统问题 → 建议联系下游系统负责人

## Notes
- 售后场景常见 Topic：
  - 退款状态变更
  - 订单状态同步
  - 申诉流程通知
- 需要根据实际情况补充具体的 CLI 工具和 API
